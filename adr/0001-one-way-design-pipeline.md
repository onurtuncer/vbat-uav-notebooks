# 1. One-way design pipeline (config → src → notebooks → out → cfd)

Status: Accepted

## Context

A conceptual design study couples many disciplines (mission sizing,
aerodynamics, structures, CAD, CFD, electrical). Left unstructured, inputs,
physics, and orchestration bleed into each other: constants get hard-coded in
notebooks, one notebook reaches into another's internals, and reproducing a
design point becomes guesswork. We need a data-flow discipline that makes the
design point reconstructible and every artifact traceable to its inputs.

## Decision

Data flows one way through five stages:

```
config/*.yaml  ->  src/conceptual_design/  ->  notebooks/  ->  out/  ->  cfd/
(inputs)           (ALL physics)               (orchestrate)   (handoffs)  (consumers)
```

- `config/*.yaml` holds every input parameter.
- `src/` holds all physics as importable modules.
- Notebooks only set parameters, call modules, interpret results, and write
  YAML/CSV **handoffs** to `out/`.
- Each notebook reads only the `out/` handoffs of the notebooks before it and
  re-runs `run_sizing_loop` from `config/` to reconstruct the same design
  point. Notebook execution order is therefore load-bearing.
- CFD and other consumers read `out/` but never feed back into it.

## Consequences

- The design point is reconstructible from `config/` alone; `out/` is a
  cache, regenerated by re-running the notebooks in order.
- Changing the sizing API means updating every notebook's `run_sizing_loop`
  call site (NB2–NB8) — a known, enumerated cost.
- Cross-notebook coupling is explicit and append-only (via `out/` files),
  which is what lets CI regenerate and regression-check the whole chain.
- No backward edges: a downstream discovery (e.g. a CG shift found in
  `mass_properties`) is fed back by changing `config/`/`src/` and re-running,
  not by editing an upstream `out/` file in place.
